# Human Request: Ajuste do Modo de IA de Destaques e Preservação de Template Modificado (req-026)

Esta especificação define o lote de trabalho `req-026` (BATCH-026) para o Engenheiro Executor. O escopo abrange duas tarefas principais:
1. **Ajuste nos Prompts de Modo de IA (Destaques e Galerias)**: Atualização das instruções de prompt em português e inglês para documentar o bloco condicional opcional `<!-- no-item < -->` no destaques, permitir variáveis adicionais nos destaques, e incluir as variáveis de link individual (`[[item#link-url]]`, `[[item#link-target]]`, `[[item#link-css-classes]]`) com a regra de âncora `<a>` obrigatória nas galerias.
2. **Preservação de Código HTML/CSS Modificado nos Editores**: Correção do bug nos módulos de **Menus**, **Galerias** e **Destaques** (Publisher Highlights) no qual, ao carregar a tela de edição ou clonagem, as modificações de HTML/CSS salvas no banco de dados são sobrescritas pela carga automática (`loadTemplate`) do modelo padrão (`template_id`).

---

## 1. Contexto e Requisitos

### 1.1 Atualização de Prompts dos Modos de IA (Destaques e Galerias)
*   **Destaques (`publisher-highlights`)**: Os prompts (em português e inglês) carecem de menção ao bloco opcional `<!-- no-item < -->`, que renderiza o estado vazio quando o destaque não tiver publicações a exibir (mecanismo processado pelo `publisher-highlights.widget.php`). Também é necessário esclarecer que o usuário pode solicitar e definir variáveis personalizadas adicionais no padrão `[[item#nome_da_variavel]]`.
*   **Galerias (`galleries`)**: Os prompts (em português e inglês) não listam as variáveis de link individual por imagem (`[[item#link-url]]`, `[[item#link-target]]`, `[[item#link-css-classes]]`) implementadas no BATCH-024. Isso faz a IA gerar o loop `<!-- item < -->` apenas com a tag `<img>` e sem a tag de âncora `<a>`. Devemos incluir a especificação destas variáveis e uma regra obrigatória de que cada imagem gerada pela IA deve estar envelopada pela tag `<a>` com os placeholders de link correspondentes.

### 1.2 Preservação de Código HTML/CSS Customizado
*   **Problema**: Nos módulos `menus`, `galleries` e `publisher-highlights`, a tela de edição/clonagem exibe no dropdown de templates (`template_id`) o modelo base selecionado. Na inicialização do DOM, os scripts JS de cada módulo identificam a presença de um valor no select do template e chamam a função `loadTemplate()`. Essa função faz uma requisição AJAX (`template-load`) e preenche os editores CodeMirror com o HTML e CSS padrões originais daquele template, deletando e sobrescrevendo as alterações salvas do registro.
*   **Sugestão de Solução**:
    *   No PHP, ao construir as opções do select de templates (`#template_id_options#`) para as telas de edição ou clonagem, se o registro possuir código HTML ou CSS customizado no banco de dados: gerar duas opções para o mesmo template: a original (ex: `<option value="menus-horizontal-navbar">Menu Horizontal</option>`) e a customizada com o sufixo `-modificado` selecionada por padrão (ex: `<option value="menus-horizontal-navbar-modificado" selected>Menu Horizontal - (Modificado)</option>`).
    *   No JavaScript, no carregamento da página:
        *   Ler o valor do select de templates. Se ele possuir o sufixo `-modificado`, **NÃO** disparar `loadTemplate()`, mantendo o código HTML e CSS injetado pelo PHP nos textareas originais do banco de dados.
        *   Guardar o HTML e CSS originais de page load em variáveis de cache cache (ex: `initialHtml` e `initialCss`).
        *   Ao detectar mudanças no select do template:
            *   Se o usuário alterar a seleção para a opção limpa (sem o sufixo `-modificado`), disparar o `loadTemplate()` via AJAX para sobrescrever os editores com o template padrão de forma dinâmica.
            *   Se o usuário retornar para a opção `-modificado`, restaurar os editores com o código customizado original armazenado no cache.
        *   Antes do envio do formulário (evento `submit`) ou da serialização (`currentSchemaOut()`), remover o sufixo `-modificado` do valor do template do input (ou do JSON de dados) para que o banco de dados armazene apenas o identificador limpo do template (evitando a gravação de strings com `-modificado` no banco de dados e mantendo a compatibilidade das queries).
