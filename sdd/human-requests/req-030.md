# Especificação de Requisitos: Autenticação Multi-Método, 2FA (App e E-mail), Login Social, Rotação JWT e Segurança (req-030)

Esta especificação define o lote de trabalho `req-030` (BATCH-030) para o Engenheiro Executor. O escopo abrange a reestruturação da autenticação administrativa do Conn2Flow, incluindo controle de métodos de login permitidos, autenticação de dois fatores (2FA) via Aplicativo Autenticador (Google Authenticator) e via E-mail, login social (OAuth 2.0 Google/Meta), rotação dinâmica de chaves JWT e endurecimento de endpoints contra Session Hijacking e CSRF.

---

## 1. Banco de Dados e Estrutura Física (Migrações Phinx)

*   **Tabela de Usuários (`usuarios`)**:
    Criar uma nova migração Phinx com timestamp atual para adicionar colunas de 2FA na tabela `usuarios`:
    *   `two_factor_secret` (varchar, 80, null): Armazena a chave secreta TOTP do usuário.
    *   `two_factor_enabled` (boolean, default false): Flag de ativação.
    *   `two_factor_type` (varchar, 20, null): Tipo de 2FA preferido (`app` ou `email`).
    *   `two_factor_email_code` (varchar, 10, null): Código 2FA temporário enviado por e-mail.
    *   `two_factor_email_expire` (datetime, null): Data/hora de expiração do código de e-mail.

*   **Tabela de Provedores Sociais [NEW] (`usuarios_provedores`)**:
    Criar uma migração Phinx para a tabela de vínculos sociais:
    *   `usuario_id` (integer, FK referenciando `usuarios.id_usuario`, ON DELETE CASCADE).
    *   `provider_name` (varchar, 30): Nome do provedor (`google` ou `meta`).
    *   `provider_uid` (varchar, 100): Identificador único retornado pelo provedor (ex: Google sub ID).
    *   `created_at` (datetime).
    *   **Índice Único**: `(provider_name, provider_uid)` para evitar vínculos duplicados.

## 2. Configurações Globais no Módulo `admin-environment`

As credenciais e permissões globais de autenticação devem ser incluídas no módulo `admin-environment` na aba **Configurações de Usuário** (`data-tab="usuario"`):

*   **Toggles de Métodos de Login**:
    *   `AUTH_METHOD_PASSWORD_ACTIVE` (Checkbox, default `true`): Permite login por Usuário/Senha tradicional.
    *   `AUTH_METHOD_GOOGLE_ACTIVE` (Checkbox, default `false`): Habilita o login pelo Google.
        *   Campos condicionalmente exibidos: `OAUTH_GOOGLE_CLIENT_ID` (text) e `OAUTH_GOOGLE_CLIENT_SECRET` (password).
    *   `AUTH_METHOD_META_ACTIVE` (Checkbox, default `false`): Habilita o login pelo Facebook/Meta.
        *   Campos condicionalmente exibidos: `OAUTH_META_APP_ID` (text) e `OAUTH_META_APP_SECRET` (password).
    *   *Nota*: Exibir em modo somente leitura (readonly) as URIs de Callback calculadas automaticamente pelo sistema:
        *   Google Redirect URI: `https://[seu-dominio]/_api/auth/callback/google`
        *   Meta Redirect URI: `https://[seu-dominio]/_api/auth/callback/meta`

*   **Controle Global de 2FA**:
    *   `AUTH_2FA_REQUIRED` (Checkbox, default `false`): Se ativo, torna a ativação e validação do 2FA obrigatória para todos os usuários que acessam o painel administrativo.
    *   **Métodos Permitidos**:
        *   `AUTH_2FA_METHOD_APP` (Checkbox, default `true`): Permite usar Aplicativo Autenticador (TOTP).
        *   `AUTH_2FA_METHOD_EMAIL` (Checkbox, default `true`): Permite usar Código enviado ao E-mail.

*   **Configurações de Rotação de Chaves JWT**:
    *   `AUTH_JWT_ROTATION_DAYS` (Integer, default `30`): Dias para expiração da chave ativa de criptografia JWT.
    *   `AUTH_JWT_GRACE_HOURS` (Integer, default `24`): Período de carência (horas) de chaves expiradas para validação retroativa de tokens de usuários ativos.
    *   **Botão "Rotacionar Chaves JWT"**: Dispara uma requisição AJAX para invalidar a chave ativa atual e gerar uma nova chave de criptografia.

*   **Persistência**: Gravar dinamicamente todas estas chaves e valores no arquivo `.env` do servidor utilizando a biblioteca existente `admin_environment_env_write()`.

## 3. Bibliotecas e Helpers (`gestor/bibliotecas/`)

### 3.1 Autenticação de Dois Fatores (`gestor/bibliotecas/2fa.php`) [NEW]
Implementar funções seguras para TOTP (RFC 6238) e e-mail:
*   `two_factor_generate_secret()`: Retorna um segredo Base32 aleatório (16 caracteres).
*   `two_factor_get_qr_code(string $email, string $secret)`: Retorna uma URI padrão `otpauth://totp/Conn2Flow:[email]?secret=[secret]&issuer=Conn2Flow` para ser renderizada em QR Code no frontend.
*   `two_factor_validate_code(string $secret, string $code)`: Retorna `true` se o código inserido bater com a janela de tempo do segredo (permitindo um drift de ±1 ciclo de 30 segundos).
*   `two_factor_email_send_code(int $usuario_id, string $email)`: Gera um número randômico de 6 dígitos, grava em `usuarios` com validade de 5 minutos, monta um e-mail HTML chamando `gestor_email_enviar()` e retorna `true`.
*   `two_factor_email_validate(int $usuario_id, string $code)`: Verifica se o código de e-mail existe, coincide com o banco de dados e está dentro do prazo de validade.

### 3.2 JSON Web Tokens com Rotação (`gestor/bibliotecas/jwt.php`) [NEW]
Implementar biblioteca de geração e validação de tokens JWT:
*   **Armazenamento de Chaves**: A lista de chaves JWT é armazenada como JSON serializado no banco de dados (tabela `variaveis`, `modulo = 'sistema'`, `chave = 'jwt_keys'`). Cada item contém: `key_id` (string/hash), `key_secret` (random hash), `created_at` (timestamp), `status` (`active` ou `expired`).
*   `jwt_generate_token(array $payload)`: Assina o token utilizando a chave do banco com status `active`.
*   `jwt_validate_token(string $token)`:
    *   Lê a assinatura do token e tenta decodificar com a chave `active`.
    *   Se falhar, lê a data de expiração da chave utilizada no token (se for uma chave `expired` criada dentro de `AUTH_JWT_GRACE_HOURS` horas, prossegue com a validação).
    *   Se a validação passar usando a chave em período de carência (grace period), retorna `['status' => 'Grace', 'payload' => $payload]`.
    *   Se passar das horas de carência ou falhar na decodificação, lança erro de autenticação.
*   `jwt_rotate_keys()`: Define a chave `active` atual como `expired`, gera uma nova chave randômica e a define como `active`. Purga registros `expired` criados além das horas de carência.

### 3.3 Integração OAuth 2.0 (`gestor/bibliotecas/oauth.php`) [NEW]
Implementar fluxo padrão de login externo:
*   `oauth_redirect_url(string $provider)`: Gera o link de redirecionamento do Google/Meta contendo `client_id`, `redirect_uri`, `scope` e `state` (hash CSRF de controle na sessão).
*   `oauth_authenticate_code(string $provider, string $code)`: Realiza a chamada POST cURL de troca de código de autorização por token de acesso e retorna o perfil verificado (ID único `uid` e `email`).

## 4. Rota de Segurança no Módulo `perfil-usuario`

Em vez de uma aba, incluir no fluxo existente do CRUD de perfil (`modulos/perfil-usuario`) o botão de ação/alteração **Segurança**:
*   Acessado via query parameter: `?configurar-seguranca=sim#formulario-alteracoes`.
*   **Interface (Fomantic UI)**:
    *   **Seção 2FA**:
        *   Se desabilitado: Exibir opções autorizadas globalmente (App e/ou E-mail). Se for App, renderizar o QR Code TOTP (usando CDN segura ou geração local) + campo de input para o código inicial de confirmação de 6 dígitos.
        *   Se habilitado: Mostrar status ativo e botão "Desativar 2FA" (exigindo senha do usuário e código 2FA atual para validar o desligamento).
    *   **Seção Contas Sociais**:
        *   Se liberado globalmente, mostrar o status dos vínculos ("Vincular Google", "Vincular Meta"). 
        *   Vínculo estabelecido exibe o ID ou E-mail da conta social associada com botão vermelho de "Desvincular".

## 5. Fluxo de Login Administrativo (`controladores/admin/login.php`)

*   **Renderização Dinâmica**:
    *   O formulário de Usuário/Senha tradicional só é renderizado se `AUTH_METHOD_PASSWORD_ACTIVE` for `true`.
    *   Os botões de login Google/Meta são renderizados condicionalmente com base em suas respectivas variáveis no `.env`.
*   **Interceptador 2FA (Fluxo Pós-Credenciais)**:
    1.  O login valida as credenciais tradicionais ou resolve o login social OAuth.
    2.  O interceptor checa se o usuário logado possui `two_factor_enabled` ativo, **ou** se `AUTH_2FA_REQUIRED` é `true` globalmente.
    3.  **Se for obrigatório e o usuário NÃO tiver 2FA ativado**:
        *   Redirecionar para uma tela intermediária de **Configuração Obrigatória do 2FA** onde o usuário deve configurar seu aplicativo autenticador ou e-mail de 2FA e validar o primeiro código antes de ter a sessão liberada.
    4.  **Se já estiver ativado**:
        *   Redirecionar para a tela intermediária de **Verificação de Segundo Fator**.
        *   Se o método preferido do usuário for E-mail, disparar imediatamente o código temporário para a caixa de entrada dele.
        *   Prender o fluxo até que o código correto seja digitado e verificado no backend.

## 6. Endurecimento de Segurança

*   **Prevenção contra Session Hijacking**:
    *   Ao autenticar, gravar na sessão do PHP `$_SESSION['client_user_agent']` e `$_SESSION['client_ip_block']` (bloco contendo os três primeiros octetos do IP do cliente, ex: `200.100.50.x`).
    *   Em todas as rotinas comuns do sistema (`gestor.php`), validar a conformidade dos dados atuais. Se houver discrepância do User-Agent ou mudança suspeita de bloco de rede, destruir a sessão do PHP imediatamente e redirecionar para a tela de login.
*   **Validação Estrita de CSRF**:
    *   O token CSRF injetado no DOM do painel administrativo deve ser gerado e validado em 100% dos controllers administrativos que recebem requisições de alteração de estado (POST/PUT/DELETE).
