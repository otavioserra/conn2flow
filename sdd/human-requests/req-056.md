# req-056: Sincronização Declarativa de Recursos, Deleção e Atualização Forçada (Módulos e Globais)

Este lote de trabalho (BATCH-056) visa generalizar e automatizar a esteira de sincronização de recursos do Conn2Flow. Permite que qualquer tabela (de módulo ou global) possa declarar regras de sincronização, conversões especiais de dados (JSON e leitura de arquivos físicos), exclusões (`deletar`) e redefinição/sobrescrita de personalizações (`forcar_atualizacao`) de forma puramente declarativa no manifesto JSON correspondente, sem necessidade de desenvolver scripts PHP de hook repetitivos.

Adicionalmente, este lote inclui a sincronização da documentação técnica correspondente em português e inglês.

---

## 1. Modificações no Compilador de Recursos (`atualizacao-dados-recursos.php`)

No arquivo `gestor/controladores/agents/arquitetura/atualizacao-dados-recursos.php`:

### 1.1 Processamento Unificado de Configurações (`gerarSchemaMetadata` e `normalizarConfigTabela`)
- Atualizar a função `normalizarConfigTabela()` e o fluxo de processamento do contrato em `gerarSchemaMetadata()` para suportar que a chave `"config"` sob `"tabela"` (em manifestos de módulos `[modulo].json`) ou sob as chaves de `"tabelas"` (no arquivo global `gestor/resources/tables_config.json`) seja tanto um **objeto associativo único** quanto um **array de objetos**.
- Se for um objeto único, convertê-lo internamente em um array de 1 elemento para manter a consistência do fluxo.
- Consolidar as chaves `"deletar"` e `"forcar_atualizacao"` configuradas para cada tabela, registrando-as de forma agregada por tabela no contrato consolidado `schema-metadata.json`.

### 1.2 Varredura e Coleta Dinâmica de Tabelas (`coletarRecursos`)
Implementar na coleta de recursos a busca de tabelas genéricas configuradas em módulos ou no global que possuem `"sync_resources": true`:
1. **Varredura de Configurações**:
   - Iterar pelas tabelas e configurações registradas em `tables_config.json` e nos manifestos de módulos `[modulo].json`.
   - Se a propriedade `"sync_resources"` estiver ativa (`true`) para uma dada configuração:
     - Resolver o nome da tabela (`tabela_nome` ou o padrão do escopo).
     - Resolver a subpasta de recursos (`resources_dir` ou o padrão igual ao nome da tabela).
     - Para cada idioma configurado (`pt-br`, `en`, etc.):
       - **Se for global**:
         - Se `metadata_file` estiver definido: ler a lista de registros do arquivo JSON externo localizado em `gestor/resources/[lang]/[metadata_file]` (ou `gestor/resources/[lang]/[resources_dir]/[metadata_file]` se `resources_dir` estiver explicitamente definido).
         - Se `metadata_file` for nulo/omitido: ler os registros inline em `gestor/resources/tables_config.json` sob a chave `"resources" -> "[lang]" -> "[tabela_nome]"`.
       - **Se for módulo**:
         - Se `metadata_file` estiver definido: ler a lista de registros do arquivo JSON externo localizado em `gestor/modulos/[modulo]/resources/[lang]/[resources_dir]/[metadata_file]`.
         - Se `metadata_file` for nulo/omitido: ler os registros inline em `[modulo].json` sob a chave `"resources" -> "[lang]" -> "[tabela_nome]"`.
2. **Processamento e Conversão de Campos**:
   Para cada registro localizado na varredura:
   - Processar os campos configurados em `"field_types"`:
     - Se o tipo for `"json"` e o valor for um array/objeto, aplicar `json_encode` com formatação adequada.
     - Se o tipo for `"file:ext"` (ex: `"file:html"`, `"file:css"`), buscar o arquivo físico `[id].[ext]` na pasta de recursos correspondente (global ou módulo) e injetar seu conteúdo como string purificada (sem BOM UTF-8) no campo especificado.
   - Resolver e preencher as colunas padronizadas (`status` = 'A', `versao`, `user_modified`).
3. **Cálculo de Checksum e Incremento de Versão**:
   - Calcular o checksum MD5 do registro baseado no payload do registro (ou nos campos mapeados como `"file:*"`).
   - Se o checksum do novo registro coincidir com o do registro existente nos dados estáticos anteriores, manter a versão anterior (`versao`). Caso mude, incrementar a versão.
4. **Acúmulo de Dados**:
   - Adicionar os registros processados ao agrupamento dinâmico de tabelas (ex: `$recursos['dynamicTablesData'][$tabela][]`).

### 1.3 Persistência de Dados Dinâmicos (`atualizarDados`)
- Na função `atualizarDados()`, iterar sobre a lista acumulada de tabelas dinâmicas em `dynamicTablesData`.
- Gerar dinamicamente o arquivo serializado `[TabelaNomePascalCase]Data.json` na pasta de destino (`gestor/db/data/`), utilizando o utilitário `dataFileNameFromTable($tabela)` para obter a nomenclatura correta (ex: `UsuariosPerfisModulosData.json` para a tabela `usuarios_perfis_modulos`).

## 3. Modificações no Atualizador de Banco (`atualizacoes-banco-de-dados.php`)

No arquivo `gestor/controladores/atualizacoes/atualizacoes-banco-de-dados.php`:

### 3.1 Sobrescrita por Atualização Forçada (`forcar_atualizacao`)
No fluxo de processamento de registros de tabelas no método `sincronizarTabela()`:
1. **Carregar Alvos de Atualização Forçada**:
   - Carregar a lista de registros configurados sob `"forcar_atualizacao"` a partir do contrato consolidado `schema-metadata.json` para a respectiva tabela.
2. **Ignorar Proteções de Projeto e Modificação do Usuário**:
   - Ao iterar e comparar os registros do banco de dados, verificar se o registro atual coincide com alguma das regras da lista `"forcar_atualizacao"` (comparando via PK ou por colunas da chave natural conforme estratégia).
   - Se coincidir:
     - **Bypass de Projeto**: Ignorar a verificação de projeto (`!$project && !empty($exist['project'])`). O registro do deploy do núcleo/sistema deve ser atualizado normalmente.
     - **Bypass de user_modified**: Ignorar a verificação `user_modified` (não preservar os campos originais do banco). O registro deve ser atualizado com todo o payload do arquivo JSON de dados.
     - **Reset de user_modified**: Se o registro no banco possuir `user_modified = 1`, forçar a alteração de `user_modified` de volta para `0` no banco (`$diff['user_modified'] = 0`), sinalizando o alinhamento com a base de código do deploy.
     - Preservar o campo `project` conforme configurado (sem alterá-lo ou limpá-lo).
