# req-088: Criação e Refinamento do Módulo "forms-search" (Formulários de Busca) e "pages-index" (Páginas Índice)

## Descrição do Problema / Contexto
Queremos expandir a plataforma adicionando a capacidade de busca textual de páginas do site. Para isso, precisamos criar dois novos módulos e seus respectivos widgets, baseados em estruturas de módulos existentes e consolidadas:

1. **`forms-search`**: Um módulo de formulários focado especificamente na captura de buscas textuais de páginas, baseado no módulo core `forms`.
2. **`pages-index`**: Um módulo de listagem/grade de páginas que exibe os resultados de busca e/ou as páginas ativas do sistema, baseado no módulo `publisher-index`.

Nesta iteração de refinamento, adaptamos o módulo `forms-search` para remover abas inúteis de formulários de contato genéricos (como email e redirecionamentos), padronizar a aba de campos adicionais ("Campos Extras"), redesenhar os 5 modelos de widget para focar puramente em barras de busca e implementar um autocomplete assíncrono completo via AJAX com paginação.

---

## Mudanças e Arquitetura Necessárias

### 1. Novo Módulo `forms-search` (Formulários de Busca)
- **Origem**: `gestor/modulos/forms/`
- **Destino**: `gestor/modulos/forms-search/`
- **Adaptações de Nomenclatura**:
  - Alterar o ID do módulo de `forms` para `forms-search`.
  - Alterar as referências de tabela de `forms` para `forms_search` e de `forms_submissions` para `forms_search_submissions`.
  - Alterar o ID numérico da tabela de `id_forms` para `id_forms_search`.
  - Mapear e atualizar arquivos e funções JavaScript/PHP internas (ex: de `forms.js` para `forms-search.js`, `forms_render` para `forms_search_render`, etc.).
- **Saneamento e Ajustes no CRUD (Aba Geral e Campos Extras)**:
  - **Remover Abas Inúteis**: O formulário de busca não envia e-mails e nem realiza redirecionamentos simples pós-sucesso. Portanto, as abas **"Email"** e **"Redirects"** devem ser completamente removidas das views de CRUD (`adicionar`, `editar`, `clonar`) e suas lógicas associadas no PHP/JS administrativos de `forms-search`.
  - **Aba Campos Extras**: Renomear a aba "Campos" para **"Campos Extras"**. Nesta aba, o usuário poderá definir campos opcionais adicionais de filtro (ex: categoria, data, tags), mas a busca básica por input de texto (`search`) é sempre obrigatória e intrínseca.
  - **Campo de Busca Padrão**: O formulário sempre renderizará o input de texto padrão de busca (`name="search"`), independentemente de haver ou não campos extras configurados.
- **Modelos Visuais Específicos (Templates)**:
  - Redesenhar os 5 templates padrão de widget (atualmente copiados de contato/newsletter) em `forms-search.json` e arquivos físicos de recursos. Eles devem ter layouts focados em barras de busca (ex: com ícones de lupa, caixa minimalista, etc.).
  - Os templates devem incluir a célula `<!-- input-search < -->` ... `<!-- input-search > -->` para demarcar onde o campo de busca padrão é injetado pelo PHP.
- **AI Mode para `forms-search`**:
  - Adaptar os prompts de IA em `forms-search.json` para que as instruções geradas para criação/edição de formulários de busca sigam este modelo sem abas de e-mail e com input de busca padrão.
- **Comportamento do Widget e Autocomplete AJAX**:
  - **Ação do Formulário**: O formulário deve usar `method="GET"`. O campo `action` aponta para o caminho configurado na aba Geral (admin). Se vazio, usa o fallback padrão para a página `pages-index/`.
  - **Autocomplete**: Adicionar suporte a autocomplete assíncrono no `forms-search.widget.js` e `.widget.php`:
    - Ao digitar no input de busca, fazer requisição AJAX para um novo endpoint no backend que consulta a tabela `paginas`.
    - Os resultados devem ser renderizados dentro de uma caixa flutuante demarcada pela célula `<!-- results-box < -->` ... `<!-- results-box > -->`.
    - Limitar a listagem inicial em 30 resultados de cada vez.
    - Se houver mais resultados, renderizar um botão "Carregar mais" no final da caixa para buscar os próximos resultados assincronamente.
    - Ao clicar em um resultado do autocomplete, o usuário é navegado diretamente para a página correspondente.
    - Se o usuário clicar no botão de submissão (lupa/buscar) ou pressionar Enter, ele é redirecionado para a página de índice completa (`pages-index/` ou o action configurado).

### 2. Novo Módulo `pages-index` (Páginas Índice)
- **Origem**: `gestor/modulos/publisher-index/`
- **Destino**: `gestor/modulos/pages-index/`
- **Adaptações de Nomenclatura**:
  - Alterar o ID do módulo de `publisher-index` para `pages-index`.
  - Alterar a referência da tabela local de `publisher_index` para `pages_index`.
  - Alterar o ID numérico da tabela de `id_publisher_index` para `id_pages_index`.
- **Comportamento Específico**:
  - **Consulta ao Banco**: Substituir o `INNER JOIN` contra a tabela `publisher_pages` por uma consulta direta à tabela **`paginas`** filtrando apenas páginas públicas e ativas:
    - Filtro obrigatório: `status='A' AND language='$language' AND tipo='pagina' AND sem_permissao=1`.
  - **Filtros e Busca**:
    - Deve interceptar a variável GET `search` (ex: `?search=termo`) na URL. Se presente, essa variável preenche inicialmente a busca textual do widget.
    - A busca textual deve filtrar a tabela `paginas` comparando contra os campos `nome` (título) e `html` (conteúdo da página), exemplo: `nome LIKE '%termo%' OR html LIKE '%termo%'`.
    - Suportar a ordenação do AJAX pelas 4 chaves existentes no módulo base: `title_asc`, `title_desc`, `date_asc`, `date_desc`.
  - **Variáveis de Item Mapeadas**:
    - No loop de renderização de itens, disponibilizar três variáveis fixas obrigatórias:
      - `[[item#title]]`: Título da página (campo `nome`).
      - `[[item#summary]]`: Resumo textual da página, obtido a partir de `strip_tags()` do campo `html` da página e truncado para ~200 caracteres (mimetizando uma descrição).
      - `[[item#url]]`: URL/caminho relativo da página (campo `caminho`).
  - **Página Padrão de Busca**:
    - Registrar uma página padrão pública com caminho `pages-index/` através do arquivo de sementes `pages-index.json`. Essa página virá pré-configurada com uma instância do widget `pages-index` para atuar como destino padrão de buscas globais disparadas em qualquer página do site.

3. **Validação Operacional**:
