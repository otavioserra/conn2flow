## Aprendizados do Compilador / Build

- **REGRA CRÍTICA — Não perder tempo com Versionamento e Checksums nos arquivos de configuração .json**:
  - **É terminantemente proibido e desnecessário** atualizar manualmente o versionamento (`version`/`versao`) ou os checksums (`checksum`) de recursos nos arquivos de configuração `.json` dos módulos (ex: `publisher-index.json`, `galleries.json`, etc.).
  - O desenvolvedor/agente executor **não deve perder tempo** com isso. Ele deve apenas registrar o novo template, página ou metadado com valores vazios para os checksums (ou mantê-los como estão) para essas chaves. Se for criar um novo recurso, use a versão inicial `1.0` e os valores dos checksums com string vazia.
  - Sempre que é feita uma atualização do sistema ou o deploy de um projeto, o seguinte arquivo é executado automaticamente e faz todas essas atualizações de versão e checksum por conta própria:
    `C:\Users\otavi\OneDrive\Documentos\GIT\conn2flow\gestor\controladores\agents\arquitetura\atualizacao-dados-recursos.php`
  - Qualquer esforço manual nessas chaves é inútil e gera ruído de diff desnecessário no controle de versão.

## Bugs Resolvidos & Lições Aprendidas

- **Variáveis `[[item#X]]` do widget de menus e o cerco de arrobas (BATCH-016)**: o template salvo no banco guarda as variáveis como `@[[item#X]]@` (cópia literal — ver memória `feedback-conn2flow-variaveis-html-paginas`), mas no fluxo de preview o html vem do editor já como `[[item#X]]` (o `template-load` converte `@[[`→`[[`). Um regex `\[\[item#X\]\]` casa apenas a parte interna e deixa sobrar `@valor@` no HTML renderizado quando a fonte é o banco. **Solução**: no renderizador (`menus.widget.php`), usar regex tolerante `/@?\[\[item#X\]\]@?/` (consome as arrobas adjacentes) e `str_replace` para `[[item#children]]`/`@[[item#children]]@`. Não remover as arrobas dos arquivos de template — só consumi-las no render.

- **Dropdown Fomantic sobre `<select>` e `.val()` (BATCH-017 item 1.3)**: quando o Fomantic converte um `<select class="ui dropdown">` via `.dropdown()`, `$('#id').val()` deixa de refletir a escolha do usuário. No `menus.js`, isso fazia `currentItemType()` cair sempre no fallback `'pagina'` (só a busca de páginas aparecia, nunca os inputs de link-custom/cabeçalho/etc.). **Solução**: ler o valor por `$('#id').dropdown('get value')` (com fallback `.val()` → default) e/ou propagar o `value` do callback `onChange(value, ...)` diretamente para o toggle.

- **Comentário HTML aninhado em componente de recurso (BATCH-017 item 1.2)**: ao documentar um componente, NÃO escrever `<!-- item -->`/`<!-- item-parent -->`/`<!-- no-item -->` dentro de um comentário HTML — comentários não aninham, o primeiro `-->` fecha o comentário externo e o restante vira texto/markup solto no DOM. Pior: `[[item#X]]` que sobrar fora de `<script>`/comentário pode ser processado como variável pelo gestor. Validar com contagem balanceada de `<!--`/`-->` e ausência de `[[item#` fora do bloco `<script type="application/json">`.

- **Simulação do html-editor para `menus` espelha a recursão do widget (BATCH-017 item 1.1)**: diferente do `publisher-highlights` (lista plana de cards replicados N vezes), a simulação de `menus` (`publisherVariablesOrSimulation`, branch `menus` em `html-editor-interface.js`) precisa reproduzir em JS a lógica recursiva de `menus.widget.php` (blocos `item`/`item-parent`/`no-item`, `[[item#children]]` recursivo, montagem da base). A massa mockada é uma **árvore JSON** no componente `html-editor-menus-simulation`; há fallback embutido no JS (`MENUS_SIM_FALLBACK`) para a simulação funcionar mesmo antes de o componente ser registrado no banco pelo pipeline.

- **AJAX público de widgets — `ajaxWidgets`, não `ajaxOpcao` (BATCH-028)**: o roteador AJAX do MÓDULO (`xxx_start()` → `switch($_GESTOR['ajax-opcao'])`) só roda quando a requisição vai para a URL do próprio módulo (painel admin). No site PÚBLICO, a requisição vai para `window.location.href` (a página publicada) e é o `gestor_pagina_widgets_ajax()` (em `gestor/gestor.php`) que processa `$_GESTOR['ajaxWidgets']` (lista de wrapper-ids separada por `<#;>`), chamando `widgets_get()` em modo AJAX → resolve `<modulo>_<func>_ajax` (ex.: `publisher_index_render_ajax`). O `grupo_slug` chega via JSON do wrapper (parâmetro `$params` da função), não por `ajaxRegistroId`. **CRÍTICO**: a função `_ajax` deve setar `$_GESTOR['ajax-json']` e **retornar string vazia** — qualquer retorno não-vazio é tratado como erro 500 por `gestor_pagina_widgets_ajax()`. O `widgetsToAjax` é montado no page load (quando `widgets_get` registra o wrapper) e exposto ao JS como `gestor.widgetsToAjax`.
- **Persistência css_compiled/html_extra_head é via html-editor, não nas views (BATCH-028)**: o componente `html-editor.html` já possui os textareas `name="css_compiled"`/`name="html_extra_head"` com os placeholders `#pagina-css-compiled#`/`#pagina-html-extra-head#` (e os `*-backup`). O html-editor (modos `editar`/`adicionarEditar`) **não** limpa esses placeholders — quem os preenche é o MÓDULO (no `$_GESTOR['pagina']`, após inserir `#html-editor#`). Antes do req-028 os 3 módulos só preenchiam `#pagina-html#`/`#pagina-css#`, deixando os de css_compiled/html_extra_head literais. Os placeholders `#html-original#`/`#css-original#` (e os novos `#css-compiled-original#`) são legado/no-op: não existem em nenhuma view — o trânsito no submit acontece pelos próprios textareas do html-editor.
- **gestor_componente() tem 2 blocos de injeção idênticos (BATCH-028)**: caminho `return_array` (lista de componentes) e caminho único. Ambos foram trocados pela helper `gestor_pagina_recursos_incluir()`. Cuidado ao editar com tabs: as linhas em branco internas das `if` carregam tabs de indentação; as linhas entre `if`s são totalmente vazias. Use `sed -n 'X,Yp' | cat -A` para conferir os bytes antes de Edit.
- **Clonar módulo do gestor por cópia + substituição (BATCH-028)**: para criar `publisher-index` (clone de `publisher-highlights`) usei `Copy-Item -Recurse` + renomeação recursiva de itens com `publisher-highlights` no nome + substituição de conteúdo. **Encoding**: usar `[System.IO.File]::ReadAllText($p,[Text.Encoding]::UTF8)` + `WriteAllText($p,$c, (New-Object System.Text.UTF8Encoding($false)))` (UTF-8 SEM BOM) — `Set-Content -Encoding UTF8` no PowerShell 5.1 injeta BOM e um BOM antes de `<?php` quebra o PHP (headers already sent). Registrar o módulo manualmente em `gestor/db/data/ModulosData.json` e `UsuariosPerfisModulosData.json` (os demais data files — Paginas/Templates/Variaveis/AlvosIa/ModosIa — são populados pelo pipeline a partir do manifest).
- **Variáveis globais de widget no template (BATCH-028)**: além de `[[item#X]]` (itens), o widget `publisher-index` resolve variáveis globais `[[grupo_slug]]`/`[[items_per_page]]`/`[[ordenacao]]` etc. nos data-attributes do contêiner via regex tolerante a arrobas `/@?\[\[chave\]\]@?/` (consome o cerco `@...@` do banco). Blocos condicionais de controle (`search-input`/`sort-select`/`load-more`) seguem o padrão de delimitadores `<!-- nome < --> ... <!-- nome > -->`: mantidos (só removendo os marcadores) ou removidos por inteiro conforme `show_*`/`tem_mais`. Use `preg_replace_callback` ao injetar conteúdo repetido para evitar que `$`/`\` sejam tratados como backreferences.

- **`linguagem_codigo` era exclusivo da tabela `variaveis` (BATCH-029)**: contra a premissa do req-029 §1 (que listava menus/galleries/publisher_highlights/publisher_index/prompts_ia/alvos_ia/modos_ia), essas 7 tabelas **já** usavam `language` nas migrações de criação. A **única** tabela que ainda usava `linguagem_codigo` era a `variaveis` (coluna real + 1488 entradas no `VariaveisData.json` + queries em `configuracao.php`/`gestor.php`/`plugins-installer.php` + lógica dual nos atualizadores). Migração feita sob aprovação (DEC-042): `renameColumn` idempotente + `replace_all` nas queries SQL. Cuidado: `$_GESTOR['linguagem-codigo']` (com hífen) e `$linguagemCodigo` (camelCase) **não** são a coluna — não trocar.

- **Contrato `schema-metadata.json` (BATCH-029)**: o gerador (`atualizacao-dados-recursos.php`) consolida `tabela.config` dos JSONs de módulo + `gestor/resources/tables_config.json` (global) num único `gestor/db/data/schema-metadata.json` (17 tabelas) lido pelo atualizador (`schemaMetadata()`). A sub-chave de regras dentro do bloco `"tabela"` chama-se **`config`** (`strategy`/`natural_key_columns`/`preserve_on_user_modified`/`insert_only`) + bloco `"deletar"`. Para regenerar só o contrato sem rodar o pipeline inteiro: `define('SDD_NO_AUTORUN', true); require gerador; gerarSchemaMetadata();`.

- **Guard `SDD_NO_AUTORUN` (BATCH-029)**: tanto o gerador quanto o atualizador rodam `main()` ao serem incluídos (o atualizador detecta inclusão por `realpath($_SERVER['SCRIPT_FILENAME']) !== __FILE__`). Para incluir em testes sem disparar a execução, ambos respeitam `if (!defined('SDD_NO_AUTORUN'))` no rodapé. O atualizador, quando incluído inline, lê `$GLOBALS['CLI_OPTS']` (setado pelo chamador `executarAtualizacaoBanco`/`api_executar_atualizacao_banco`).

- **Detecção de duplicata portável MySQL/SQLite (BATCH-029)**: MySQL emite `Duplicate entry`, SQLite emite `UNIQUE constraint failed`. No fallback individual do loteador `inserirEmLote`, detectar ambos para que os testes com SQLite passem e o skip de duplicata funcione em prod (MySQL). `$pdo->exec(...)` é método PDO (não shell) — não confundir com `exec()` ao grepar subprocessos.

- **Edição concorrente de agentes (BATCH-029)**: outro agente tocou `atualizacoes-banco-de-dados.php`, `atualizacao-plugin-banco-de-dados.php` e `plugins-installer.php` adicionando `log_unificado()`/`EXTERNAL_LOGGER` (Slice 5) enquanto eu refatorava o Slice 4. As edições foram **incorporadas sem reverter** (decisão do Chefe). O `passthru()` remanescente em `atualizacoes-sistema.php` é o auto-bootstrap do script de deploy (não roda atualização de banco) — **não** é alvo da remoção de CLI exec.

- **`banco.php` só implementa `mysqli` (BATCH-030)**: apesar de o `tests/bootstrap.php` definir `$_BANCO['tipo']='sqlite'`, a `banco.php` só trata `mysqli` (não há branch PDO/SQLite). Logo, funções `banco_*` em teste exigem MySQL real (gated por `CONN2FLOW_RUN_DB_TESTS=1`/`CONN2FLOW_TEST_BOOTSTRAP_CONFIG`). Para validar lógica que depende de `banco_*` em unit test, use **stubs em memória** das funções (`banco_select`/`banco_update_campo`/`banco_update_executar`/`banco_insert_name_campo`/`banco_insert_name_campos`/`banco_insert_name`) num script standalone — não dá para redeclarar as reais dentro do PHPUnit (bootstrap já carrega `banco.php`).

- **Padrão de buffer do `banco.php` (BATCH-030)**: update = `banco_update_campo('col',$v)` (acumula em `$_BANCO['update-campos']`) → `banco_update_executar('tabela',"WHERE ...")`. Insert = `banco_insert_name_campo('col',$v)` → `banco_insert_name(banco_insert_name_campos(),'tabela')`. Select = `banco_select(['unico'=>true,'tabela'=>...,'campos'=>[...],'extra'=>"WHERE ..."])`. PK de `usuarios` é **`id_usuarios`** (plural), não `id_usuario`.

- **Função de e-mail real é `comunicacao_email()` (BATCH-030)**: vários intakes citam `gestor_email_enviar()` (não existe). A real está em `bibliotecas/comunicacao.php`: `comunicacao_email(['destinatarios'=>[['email'=>,'nome'=>]], 'mensagem'=>['assunto'=>,'html'=>]])`, lê SMTP de `$_CONFIG['email']` e retorna booleano. Para forçar config em teste, passar `EMAIL_TESTS`+`EMAIL_HOST/USER/PASS/...`.

- **URL base e `.env` em runtime (BATCH-030)**: `$_GESTOR['url-full-http']` = `https://{SERVER_NAME}{url-raiz}` (definido em `config.php`); usei para montar o callback OAuth `/_api/auth/callback/{provider}`. Variáveis do `.env` chegam em runtime via superglobal `$_ENV[...]` (sem precisar `global`); é assim que `admin-environment` lê/grava (`admin_environment_env_read`/`admin_environment_env_write`, que reescreve linhas do `.env` em `$_GESTOR['AUTH_PATH_SERVER'].'.env'`).

- **Timestamps de migração Phinx (BATCH-030)**: o projeto usa datas crescentes arbitrárias (algumas "no futuro": `20260701*`, `20260705100000` era o maior). Para uma migração nova rodar **por último**, use timestamp > o maior existente (usei `20260706100000`/`20260706100010`). Migrações de ALTER devem ter guards `hasTable()`/`hasColumn()` + `up()`/`down()` idempotentes (padrão de `20260611110000_add_css_compiled...`).

- **2FA/TOTP próprio validado por vetores RFC (BATCH-030)**: `bibliotecas/2fa.php` implementa Base32/HOTP/TOTP byte-wise (sem `mb_*` → roda mesmo sem mbstring). Vetores de regressão: secret ASCII `12345678901234567890` ↔ Base32 `GEZDGNBVGY3TQOJQGEZDGNBVGY3TQOJQ`; HOTP counter 0→`755224`, 1→`287082` (RFC 4226 Apêndice D); TOTP T=59→`287082`, T=1111111109→`081804` (RFC 6238). `jwt.php` assina HS256 com `kid`; chaves em `variaveis` (`modulo='sistema'`, `id='jwt_keys'`, `valor` JSON).

- **Registrar bibliotecas novas em `config.php` (BATCH-030 Slices 3-6)**: `gestor_incluir_biblioteca('x')` resolve via `$_GESTOR['bibliotecas-dados']['x']` (mapa nome→arquivos em `gestor/config.php`, ~linha 266). Toda lib nova (`2fa`/`jwt`/`oauth`/`seguranca`) precisa ser registrada lá, senão `gestor_incluir_biblioteca` é no-op silencioso. Chave pode começar com número (`'2fa' => Array('2fa.php')`).

- **Roteamento por página exige registro no manifest (BATCH-030 Slice 5)**: o roteador do gestor mapeia URL `path` → módulo + `option` pelas páginas registradas (tabela `paginas` populada a partir do `*.json` do módulo). Toda **nova opção** no `switch($_GESTOR['opcao'])` (ex.: `signin-2fa`, `social-login`, `oauth-callback`) precisa de uma página correspondente em `resources/{lang}/pages/<id>/<id>.html` **e** registro no array `pages` do JSON (com `path`, `option`, `layout`, `without_permission`). Sem isso a URL dá 404. Páginas que só redirecionam (header+exit) ainda precisam de um HTML mínimo.

- **`modelo_tag_in($pagina,'<!-- x < -->','<!-- x > -->',$valor)` (BATCH-030)**: substitui o **bloco inteiro** (marcadores incluídos) por `$valor`. Passar `''` remove o bloco; passar HTML injeta no lugar. É o mecanismo usado pelo `perfil_usuario_editar` para mostrar/ocultar blocos `*-campos`. Usei para a seção `seguranca-campos` e o `login-senha`.

- **Ponto de Session Hijacking = `gestor_permissao_token()` (BATCH-030 Slice 6)**: em `gestor/gestor.php` (≠ `bibliotecas/gestor.php`), essa função valida o token a cada requisição e retorna `true` após setar `$_GESTOR['usuario-id']`/`usuario-token-id`. Inseri a validação UA+bloco-IP imediatamente antes do `return true` (fail-safe). **Importante**: registrar os marcadores (`seguranca_sessao_registrar`) só no **login efetivo** (signin inline, `finalizar_login`), **nunca** em `usuario_gerar_token_autorizacao` — esta é chamada na renovação automática de token dentro do próprio `gestor_permissao_token`, e re-gravar o UA atual a cada renovação anularia a proteção.

- **Sessão: SEMPRE usar o store do sistema, nunca `$_SESSION` nativo (BATCH-030)**: o sistema tem uma seção "Funções de Sessões e Cookies" em `bibliotecas/gestor.php` (`gestor_sessao_variavel` get/set, `gestor_sessao_variavel_del`, `gestor_sessao_del`, `gestor_sessao_del_all`) persistida em `sessoes_variaveis` (cookie `session-authname`), **iniciada automaticamente** por `gestor_sessao_iniciar()` no `gestor/gestor.php` → disponível em qualquer módulo/controlador sem `session_start()`. Get retorna `''` quando ausente (use `existe()` ou `(int)` para checar). Usei esse store para CSRF token, marcadores de hijacking E todo o estado de fluxo 2FA/OAuth (`pending_2fa_*`, `oauth_state/provider/action/link_user`). **A primeira versão usou `$_SESSION` nativo e foi corrigida por feedback do Chefe** — não misturar os dois sistemas de sessão.

- **admin-environment `.env` (BATCH-030 Slice 3)**: o padrão é `admin_environment_env_read()` (lê `$_ENV` com defaults) → `$dados` → `modelo_var_troca('#placeholder#', ...)`; o save coleta `$_REQUEST` em `$data` e chama `admin_environment_env_write($data)` (reescreve linhas do `.env`). Toggles no front enviam `'true'/'false'` (string) via `.checkbox('is checked')`. Para campos novos: adicionar nas 4 pontas (env_read, raiz/troca, ajax_salvar, HTML+JS).

- **`VariaveisData.json` é COMPILADO de `variables.json` (BATCH-034)**: nunca editar `gestor/db/data/VariaveisData.json` à mão — ele é gerado a partir de `gestor/resources/{pt-br,en}/variables.json` pelo pipeline (`Update => Core`). Adicionar variável só no `variables.json` de cada idioma (schema `{id, value, type}`); o compilado (`{language, modulo, id, valor, tipo, grupo, descricao}`) sai automático no deploy. (Orientação do Engenheiro Chefe Humano.)

- **Arquitetura do Editor HTML Visual (BATCH-034)**: três camadas — (1) `assets/interface/html-editor.js` roda DENTRO do iframe de preview (injetado no `srcdoc` por `editorHtmlVisualConteudo()` em `html-editor-interface.js`); (2) `html-editor-interface.js` orquestra na JANELA PAI (monta o srcdoc, trata save/screenPagina/imagepick); (3) `html-editor-helper.js` roda na página principal do gestor e clona `#html-editor-modal` + expõe `variablesEnvironment()` (`htmlEditorModalHtml`/`htmlEditorVars`/`htmlEditorScriptPath`) que o interface.js injeta no srcdoc. O modal de edição (`#html-editor-modal`, modos text/image/code) vive DENTRO do iframe (não no pai). Comunicação pai↔iframe por `postMessage`: ImagePick usa `html-editor-imagepick-open/selected`; o req-034 adicionou o namespace `c2f-he:*` (`undo/redo/insert-element/insert-widget/cancel-insert/history`). O novo `html-editor-visual-controls.js` (janela pai) foi criado para não inflar o interface.js (3236 linhas).

- **z-index: overlays do iframe cobrem o modal Fomantic (BATCH-034)**: os overlays/toolbar do editor visual têm z-index altíssimo (~999990) para ficar acima do conteúdo; mas o modal de edição Fomantic (também no iframe) tem z-index ~1000. Solução: no `onShow` do modal, esconder o "chrome" (selection overlay/toolbar/breadcrumb/styler) e restaurar no `onHide` (`updateSelectionUI`). Só `hideHover()` não basta. **Bug do click residual pós-DnD**: o `mouseup` que finaliza um arraste gera um `click` que re-selecionaria outro elemento — usar flag `suppressClick` consumida no listener de click (capture).

- **Comentários de widget e tabelas de widget (BATCH-034)**: o roteador em `gestor/gestor.php` casa wrappers com `/<!--\s*widgets#(.+?)\s*<\s*-->([\s\S]*?)<!--\s*widgets#\s*\1\s*>\s*-->/i`; a assinatura `\1` é `MODULO->FUNCAO(JSON)`. Os 4 widgets do sistema (`menus`, `galleries`, `publisher-highlights`, `publisher-index` — com hífen) usam `->render({"grupo_slug":"<slug>"})`. As 4 tabelas têm colunas `id` (slug), **`name`** (nome — não `nome`), `language` e `status` (`'A'`). A rota AJAX `html-editor-widgets-list` filtra `status='A' AND language=<idioma>` e devolve `{id, nome}` por módulo. O save do iframe expõe `window.htmlEditorGetCleanHtml()` (remove UI + reconverte `.conn2flow-widget-wrapper`→comentários); o `previsualizarConfirmar/Voltar` do interface.js usa essa função com fallback. O dimmer/loader do preview deve permanecer irmão do `#iframe-preview` (o interface.js usa `iframe.parent().find('.ui.dimmer')`).

- **`banco_select` mapeia chave de retorno pela string literal do campo (BATCH-041)**: `banco_select` (em `bibliotecas/banco.php`) monta o array associativo de retorno fazendo `explode(',', $camposVirgulas)` e casando posicionalmente com `banco_row_array`.
