Credenciais dev: `root`/`root123`, app `conn2flow_user`/`conn2flow_pass`, banco real `conn2flow`. **Para testes, criar um banco dedicado `conn2flow_test` (DROP ao fim) — nunca tocar `conn2flow`.** Padrão usado: `define('SDD_NO_AUTORUN',true); require atualizador;` → setar `$GLOBALS['LOG_FILE_DB']`/`$GLOBALS['CLI_OPTS']=['orphans-mode'=>'ignore']` → criar tabelas mínimas e chamar `sincronizarTabela()`/`executarDelecoes()` direto com um PDO próprio. - **Container vs workspace (BATCH-029)**: `conn2flow-app` monta `dev-environment/data/sites` (o **mirror** sincronizado) em `/var/www/sites` e `.../public_html` em `/var/www/html` — **não** monta o workspace `gestor/`. - **MSYS path mangling (Git Bash + docker exec)**: `docker exec ... php /tmp/x.php` faz o Git Bash traduzir `/tmp/...` para um path Windows (`C:/Users/.../Temp/...`). Prefixar com `MSYS_NO_PATHCONV=1` ou envolver em `sh -lc 'php /tmp/x.php'`.

## Aprendizados do Compilador / Build

- **REGRA CRÍTICA — Não perder tempo com Versionamento e Checksums nos arquivos de configuração .json**:
  - **É terminantemente proibido e desnecessário** atualizar manualmente o versionamento (`version`/`versao`) ou os checksums (`checksum`) de recursos nos arquivos de configuração `.json` dos módulos (ex: `publisher-index.json`, `galleries.json`, etc.).
  - O desenvolvedor/agente executor **não deve perder tempo** com isso. Ele deve apenas registrar o novo template, página ou metadado com valores vazios para os checksums (ou mantê-los como estão) para essas chaves. Se for criar um novo recurso, use a versão inicial `1.0` e os valores dos checksums com string vazia.
  - Sempre que é feita uma atualização do sistema ou o deploy de um projeto, o seguinte arquivo é executado automaticamente e faz todas essas atualizações de versão e checksum por conta própria:
    `C:\Users\otavi\OneDrive\Documentos\GIT\conn2flow\gestor\controladores\agents\arquitetura\atualizacao-dados-recursos.php`
  - Qualquer esforço manual nessas chaves é inútil e gera ruído de diff desnecessário no controle de versão.

## Bugs Resolvidos & Lições Aprendidas

- **Arquitetura de janelas do Live Editor (BATCH-082 §1)**: no Live Editor, `dashboard.toolbar.js` roda na PÁGINA HOSPEDEIRA (top window) e o motor `html-editor.js` é carregado na MESMA window (via `loadScriptOnce(getRaiz()+'interface/html-editor.js')` → `instantiateEditor`). A "toolbar" é um IFRAME injetado na página (`dashboard.iframe-toolbar.js`), que posta **objetos** `c2f-toolbar:*` para `window.parent` (= a página). O motor, ao inserir widget, chama `requestWidgetRender` → posta **string JSON** `c2f-he:widget-render` para `window.parent` (na top, `window.parent===window`) e escuta `c2f-he:widget-rendered` via `JSON.parse(e.data)`. **Consequência p/ novos listeners no toolbar.js**: o listener principal faz `data=ev.data||{}` e filtra por `data.type` (objetos do iframe) — mensagens **string** do motor caem no `!data.type` e são ignoradas. Solução: **listener dedicado** que só trata `typeof ev.data==='string'` + `JSON.parse` + `data.action`. A rota AJAX nova (`site-toolbar-widget-render`) apenas delega a `html_editor_ajax_widget_render()` (já existente na lib, resolve `modulo->func({...})` em modo page-load com `$_GESTOR['ajax']=false`). Filtrar `action==='c2f-he:widget-render'` evita laço com a própria resposta `-rendered` postada na mesma window.
- **Restauração de backup no Live Editor precisa espelhar o `site-toolbar-render`, não injetar caixas (BATCH-082 §3)**: o bug era `restorePageBackup` injetar o `html` renderizado com `.c2f-widget-box` (widget "morto", não interativo) e sem re-mapear. O `mapTree` (BATCH-077/079) exige que o DOM vivo tenha os **comentários** `<!-- widgets#SIG < --> … > -->` (delimita/anota o widget) e as variáveis **resolvidas em texto puro** (ele as re-envolve em `.c2f-var-box`). Por isso: (a) `dashboard_site_toolbar_boxes_widgets` passou a emitir os comentários (não `.c2f-widget-box`); (b) criado `dashboard_site_toolbar_resolver_variaveis` (valor puro, SEM box) — usar `.c2f-var-box` no `html` faria o `mapTree` **duplicar** a variável (cria box vazia ao lado da existente, pois `annotateTextVars` casa o template contra o text node); (c) `backup-get` passou a devolver `raw` (cru, marcadores intactos) e o frontend injeta `html` no DOM vivo + `raw` no container oculto (`#c2f-raw-content` p/ conteúdo; backup inteiro p/ layout, com slot `#c2f-page-content` vazio → o conteúdo vivo já anotado NÃO é reprocessado) e re-roda `mapTree`+`restoreChrome`. **NÃO zerar `varMap`** (o intake pedia `varMap={}`): ao restaurar layout, o conteúdo vivo não é re-mapeado (slot vazio) e o reset apagaria seu mapeamento, queimando variáveis no save; preservar deixa só lixo inofensivo (reconstructOriginal ignora IDs órfãos). O motor legado de caixas do `render` (`html`/`layout_html`) NÃO é consumido pelo frontend (usa `content_raw`/`layout_raw`), então mudar `boxes_widgets` não gerou regressão.
- **Isolamento de projeto no core = HOOK, nunca chamar `multiusuario_*` direto (BATCH-082 §4 — corrigido pelo Chefe)**: regras específicas de projeto (multi-usuário) NÃO podem viver no core (`conn2flow`). O core **expõe um ponto de extensão** via a lib `gestor/bibliotecas/hooks.php` (estilo WordPress): `hook_apply_filters($namespace,$evento,$value,...$args)` (retorna o valor filtrado; callbacks recebem `($value,...$args)`) ou `hook_do_action(...)`. O projeto **consome** registrando o callback em `gestor/project/hooks/hooks.json` → `controllers[<namespace>]=<arquivo.hooks.php>` (resolve o arquivo do handler p/ aquele namespace) + `filters[<namespace>][<evento>]=<funcao>` (ou `actions`). O pipeline `hooks_registrar_projeto()` sincroniza o JSON → tabela `hooks` no **deploy do projeto** (sem deploy, o filtro não é carregado em runtime; `ensureLoaded` lê da tabela por namespace/evento com lazy-load + require_once do controller). Neste batch: core `dashboard_site_toolbar_verificar_permissao_pagina($pagina)` = `return hook_apply_filters('dashboard','site-toolbar.permissao-pagina', true, $pagina)` (guard `function_exists('hook_apply_filters')`, default permissivo); handler `multiusuario_dashboard_site_toolbar_permissao_pagina_filter($permitido,$pagina)` no `conn2flow-site/.../multiusuario.hooks.php`. Operações do projeto: `acesso-completo-a-paginas`/`admin-paginas` e `acesso-completo-a-paginas-publisher`/`publisher-pages`; usuário logado via `gestor_usuario()['id_usuarios']`. **Erro cometido e corrigido**: a 1ª versão chamou `multiusuario_tem_acesso_completo` dentro do `dashboard.php` (core) com guard `function_exists` — o Chefe apontou o acoplamento indevido; o correto é core-emite-hook / projeto-consome. Exemplos vivos de filtro core→projeto: `hook_apply_filters('html-editor','templates.load.where',$where)`, `hook_apply_filters('ia','prompts.load.where',$where)`.
- **Testar funções internas de IIFE (dashboard.toolbar.js)**: `tests/Unit/JS/dashboard.toolbar.test.js` injeta um hook `window.__c2fToolbar={...}` **antes** do fecho `})();` (via `code.lastIndexOf('})();')` + splice) expondo as funções internas (`runMap`/`reconstruct`/`handleWidgetRender`/`restorePageBackup`). Para testar a ponte de widget, mockar `globalThis.fetch` e espionar `window.postMessage` (a função `handleEngineWidgetRender` **retorna a promise** do fetch p/ permitir `await`). `c2fEditor` é null no teste → os guards `typeof c2fEditor!=='undefined' && c2fEditor && ...` degradam graciosamente (deselectAll/restoreChrome viram no-op). Testes do motor (`insertTemplate`) ficam em `html-editor.live.test.js` (carrega `html-editor.js` via `vm.runInThisContext` + `__c2fHtmlEditorNoAutoInit`).
- **CAUSA-RAIZ DEFINITIVA do "302 → home" no save do site toolbar (BATCH-081) = `interface_modulo_variavel_valor` redireciona fora do CRUD admin**: confirmada por instrumentação com backtrace (`gestor_redirecionar` no mirror). Cadeia: `dashboard_ajax_site_toolbar_save` → `interface_historico_incluir()` → `interface_modulo_variavel_valor()` → `gestor_redirecionar_raiz()` → `header('Location: …/dashboard/'); exit`. O `interface_historico_incluir`, quando **não** recebe `id_numerico_manual` (nem `sem_id`), resolve o id numérico via `interface_modulo_variavel_valor(['variavel'=>tabela.id_numerico])`, que **exige `$_GESTOR['modulo-registro-id']`** (vem de `$_REQUEST['ajaxRegistroId']`, presente só no CRUD admin). No AJAX do site toolbar esse id não existe → a função redireciona à raiz do módulo. Como `gestor_redirecionar` **não checa `$_GESTOR['ajax']`** (só faz `header('Location')`+`exit`), o `fetch` recebe **302 `text/html`** e segue para a home — sem erro no log do PHP. **Intermitente** porque `interface_historico_incluir` só roda quando o conteúdo (`$algum`) ou o layout (`$layoutSalvo`) muda de fato. **CORREÇÃO**: passar `id_numerico_manual` (`$pagina['id_paginas']` / `$layout['id_layouts']`, já disponíveis) nas **duas** chamadas de `interface_historico_incluir` (página em `dashboard_ajax_site_toolbar_save`, layout em `dashboard_site_toolbar_salvar_layout`) — assim usa o id conhecido e nunca chama a função que redireciona. **Regra geral**: ao reusar `interface_historico_incluir`/`interface_backup_campo_incluir` (e afins) FORA do fluxo CRUD admin (ex.: AJAX do dashboard), SEMPRE passar `id_numerico_manual`/`sem_id`, senão `interface_modulo_variavel_valor` derruba a requisição com um redirect.
- **Hipóteses DESCARTADAS nesta investigação (registrar p/ não repetir)**: (1) **NÃO** era warning/notice poluindo o JSON — o log PHP estava vazio; uma blindagem `ob_start`/`ob_end_clean` em `gestor.php` foi tentada e **revertida**. (2) **NÃO** era `request_order` sem "P": este servidor tem `request_order => no value` → PHP usa o fallback `variables_order = EGPCS`, que **inclui POST** em `$_REQUEST` (confirmado por `php -i`). Logo o roteamento por query string (`ajax`/`ajaxOpcao` na URL) e o `array_merge($_REQUEST,$_POST)` em `dashboard_start` **não eram necessários** — foram mantidos como convenção **defensiva** (robustez caso outro ambiente exclua POST de `$_REQUEST`), com comentários honestos apontando o id_numerico_manual como o fix real. Lição: instrumentar com **backtrace no ponto do `header('Location')`** resolve esse tipo de "302 fantasma" em minutos — não teorizar sobre config de PHP sem medir.
- **Loop de feedback do `ResizeObserver` em caixa com `height:auto` (BATCH-081 §2 feedback)**: observar um contêiner `resize:both` e, no callback, ajustar a altura de um filho (textarea/CodeMirror) que **influencia** a altura do contêiner (quando ele é `height:auto`, dirigido pelo conteúdo, com `flex` + `max-height`) gera oscilação de altura ao abrir (cresce até bater no `max-height` e para). Correção: (1) dar **altura inicial DEFINIDA** ao contêiner (`.c2f-he-modal-box` → `height:70vh`; `#c2f-ai-panel/.#c2f-custom-panel .c2f-he-live-box` → `height:70vh`) para desacoplar a altura do conteúdo; (2) **guarda anti-loop** no callback (arredondar e só aplicar quando o valor muda: `if(el._c2fH!==h){…}` / `if(cmEl._c2fAvail===avail)return;`). CodeMirror não acompanha a altura do container por CSS — precisa de `setSize('100%', h)` no observer; a instância fica em `element.CodeMirror` (API CM5).
- **Save do Live Editor: corrida "Editar → Salvar imediatamente" (BATCH-081)**: `startEdit` é assíncrono (fetch render → `mapTree` → carga do `html-editor.js` → `instantiateEditor` com retries). Correção: guarda de prontidão no `saveEdit` (aborta com aviso se `c2fEditor`/`getCleanHtml` ausentes) + `deselectAll()` determinístico antes do `getCleanHtml`.
- **CodeMirror no Live Editor usa o loader do `dashboard.toolbar.js` (BATCH-081)**: no live editor o CodeMirror é carregado via `ensureCodeMirror` (CDN 5.65.20) com um subconjunto de modos. Para `mode:'markdown'`/`'htmlmixed'` funcionarem, incluir `mode/markdown/markdown.js` e `mode/javascript/javascript.js` na `loadScriptsSeq`. No motor (`html-editor.js`) instanciar só com `typeof CodeMirror !== 'undefined'` (degrada para textarea) e ler/escrever via `getValue()/setValue()` (CM5 os expõe diretos, além de `getDoc()`). - **Variável de template `@[[usuario#nome]]@` (BATCH-081)**: é resolvida em `gestor/gestor.php` (`modelo_var_troca_tudo(..., 'usuario#nome', $usuario['nome'])`), disponível em qualquer página renderizada pelo pipeline (inclui o `dashboard-site-toolbar`). - **BATCH-049 Forms + html-editor**: para transformar `forms` em widget visual sem criar tabelas auxiliares, manter `fields_schema` como contrato único (`form_action`, limites, reCAPTCHA, `email`, `redirects`, `fields[]`, `template_id`) e adicionar apenas colunas de layout (`html`, `css`, `css_compiled`, `html_extra_head`). O alvo `forms` precisa ser tratado como família `[[item#X]]` em `html-editor.php`, `html-editor-interface.js` e `html-editor-modules.js`; o renderer público processa `<!-- item < --> ... <!-- item > -->` e blocos condicionais `type-input`/`type-textarea`/`type-select`. O widget público usa preferencialmente `forms->render({"form_id":"slug"})`, aceitando `grupo_slug`/`id` como fallback.
- **BATCH-050 Forms no html-editor**: quando o HTML salvo pode conter tags como `<textarea>`, preencher o textarea do editor com valor bruto quebra o DOM. A correção deve ficar no componente `html_editor_componente()` via parâmetro opcional `html_specialchars`, escapando apenas `#pagina-html#` e `#pagina-html-extra-head#` com `htmlspecialchars(..., ENT_QUOTES, 'UTF-8')`; o submit/banco continuam trabalhando com HTML bruto. Para abas Fomantic aninhadas no Forms, usar `context: 'parent'` no menu externo e `context: '[data-tab="forms-template"]'` no menu interno evita a sub-aba "Widget" esconder a aba pai. Novos templates Forms seguem o padrão físico `resources/{lang}/templates/<id>/<id>.html`, manifest com `version: "1.0"` e checksums vazios antes do compilador.
- **BATCH-051 Forms opções/radio/checkbox**: o escape do editor agora deve ser ativado pela presença de `html`/`html_extra_head` nos params, sem depender de `html_specialchars`. No renderer de Forms, `options` aceita linhas com `value:label` ou `value|label`; sem separador, mantém valor=label. `radio`/`checkbox` reaproveitam `options` e blocos `type-radio`/`type-checkbox`; checkbox usa `name[]`. - **Sincronização declarativa de tabelas — gerador + atualizador (BATCH-056)**: o motor do BATCH-029 foi generalizado. `normalizarConfigTabela()` em `atualizacao-dados-recursos.php` agora retorna **lista** de configs (era `?array` de 1): `config` pode ser objeto único (→ array de 1) ou array de objetos, cada um podendo nomear sua tabela via `tabela_nome`. Novo `coletarConfigsTabelas()` é o ponto único de coleta (global `tables_config.json` + blocos `tabela.config` dos módulos, ordem núcleo→global→módulos) reaproveitado por `gerarSchemaMetadata()` (contrato) e `coletarRecursos()` (varredura dinâmica). O contrato `schema-metadata.json` ganhou o mapa de topo **`forcar_atualizacao`** (espelha `deletar`, agregado por tabela); os campos de varredura (`sync_resources`/`resources_dir`/`metadata_file`/`field_types`/`scope`/`base_dir`/`inline`) são build-time e **não** entram no contrato. Regenerar via `define('SDD_NO_AUTORUN',true); require gerador; gerarSchemaMetadata();` preserva as 17 tabelas.
- **Varredura `sync_resources` (BATCH-056)**: helpers novos no gerador — `lerMetadadosDinamicos()` (externo via `metadata_file` ou inline em `resources->lang->tabela`; caminho do módulo sempre usa subpasta `resources_dir|tabela`, global usa subpasta só se `resources_dir` explícito), `processarRegistroDinamico()` (`field_types`: `json`→`json_encode`, `file:<ext>`→conteúdo de `<id>.<ext>` sem BOM via `readFileIfExists`; injeta `language`/`status='A'`, e `module`/`modulo` só quando estão na natural key), `checksumRegistroDinamico()` (MD5 dos arquivos `file:*` quando há, senão do payload). `coletarRecursos` acumula em `dynamicTablesData[tabela][]` e `atualizarDados` grava `<Pascal>Data.json` via `dataFileNameFromTable`, **pulando as 9 tabelas fixas reservadas** (layouts/paginas/componentes/templates/variaveis/prompts_ia/modos_ia/alvos_ia/forms) para não colidir com a coleta clássica.
- **`forcar_atualizacao` no atualizador (BATCH-056)**: em `sincronizarTabela()` há **3 caminhos de update** (PK, chave natural principal, fallback de linguagem ausente) — o bypass precisa ser aplicado nos três. Padrão: helper `$isForced($exist)` (casa por `$pkDeclarada` em `$forcedPks` e/ou por `naturalKeyGenerica($exist)` em `$forcedNaturais`); quando forçado: guardar a proteção de projeto com `&& !$forced`, o skip de `user_modified` no loop de diff com `&& !$forced`, a preservação `preserve_on_user_modified` com `&& !$forced`, **adicionar `$diff['user_modified']=0`** se era 1 (só se a coluna existe em `allowedCols`), e **NÃO** marcar `project` (preserva o existente). `schemaMetadata()` passou a ler a chave `forcar_atualizacao`.
- **Testar `sincronizarTabela` com SQLite (BATCH-056)**: dá para passar um `PDO('sqlite::memory:')` direto (a função recebe `$pdo`, não usa `banco.php`/mysqli). `SHOW COLUMNS`/`SHOW VARIABLES` lançam exceção no SQLite mas são tratadas (catch) → `allowedCols=null` (sem filtro de coluna) e packet usa fallback. Apontar o contrato para um `schema-metadata.json` temporário **antes** da 1ª chamada a `schemaMetadata()` (cache estático). Definir `id_numerico` + PK no contrato faz o UPDATE usar `WHERE pk=:__pk` e evitar o operador `<=>` (que é MySQL, não SQLite). Incluir o arquivo no PHPUnit exige `define('SDD_NO_AUTORUN', true)` antes do `require_once` (o guard impede o autorun de `main()`); o bootstrap já carrega `gestor.php`/`banco.php`, então o `require_once` do updater não duplica. `log_unificado` faz `echo` em CLI → envolver a chamada de sync em `ob_start()/ob_end_clean()` no teste.
- **Deprecation `${var}` (PHP 8.4)**: `atualizacoes-banco-de-dados.php` tinha `"~${totalUpd}"` que só virou warning visível quando o `ForcarAtualizacaoTest` passou a incluir o arquivo no PHPUnit. - **Variáveis `[[item#X]]` do widget de menus e o cerco de arrobas (BATCH-016)**: o template salvo no banco guarda as variáveis como `@[[item#X]]@` (cópia literal — ver memória `feedback-conn2flow-variaveis-html-paginas`), mas no fluxo de preview o html vem do editor já como `[[item#X]]` (o `template-load` converte `@[[`→`[[`). Um regex `\[\[item#X\]\]` casa apenas a parte interna e deixa sobrar `@valor@` no HTML renderizado quando a fonte é o banco. **Solução**: no renderizador (`menus.widget.php`), usar regex tolerante `/@?\[\[item#X\]\]@?/` (consome as arrobas adjacentes) e `str_replace` para `[[item#children]]`/`@[[item#children]]@`.
