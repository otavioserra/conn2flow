# Signup Hooks in the perfil-usuario Module

## Overview

Starting with Host Manager v0.3.0, the `perfil-usuario` (core) module gained **two extension points** in the `perfil_usuario_signup()` function — an action hook and a filter hook. They allow any downstream module or project to intercept the new user registration process to execute additional logic (account creation, resource provisioning, etc.) and, optionally, change the post-signup redirect URL.

No existing functionality was altered. The hooks are **transparent** — if no callbacks are registered, behavior is identical to before.

---

## Added Hooks

### Action: `perfil-usuario/signup.banco`

```php
gestor_incluir_biblioteca('hooks');

hook_do_action('perfil-usuario', 'signup.banco', $id_usuarios, [
    'nome'   => $nome,
    'email'  => banco_escape_field($_REQUEST['email']),
    'id'     => $id,
    'plano'  => $_REQUEST['plano'] ?? null,
    'domain' => $_REQUEST['domain'] ?? null,
]);
```

**Position in code:** Immediately after `$id_usuarios = banco_last_id()` — the user already exists in the database with a valid ID.

**Position in flow:**
```
Validation → reCAPTCHA → Create User → banco_last_id()
    ↓
    ★ hook_do_action('perfil-usuario', 'signup.banco', ...)
    ↓
Plans (disabled) → Hosts (disabled) → Login → Token → Email → Redirect
```

**Arguments passed:**

| Arg | Type | Description |
|-----|------|-------------|
| `$id_usuarios` | int | Numeric ID of the newly created record |
| `$dados['nome']` | string | Full name (already sanitized) |
| `$dados['email']` | string | Email (sanitized via `banco_escape_field`) |
| `$dados['id']` | string | Slug generated by `banco_identificador()` |
| `$dados['plano']` | string\|null | `$_REQUEST['plano']` parameter (may be null) |
| `$dados['domain']` | string\|null | `$_REQUEST['domain']` parameter (may be null) |

**Intended use cases:**
- Create a hosting account linked to the new user
- Register the user in external systems (CRM, mailing list)
- Send internal notifications to admins
- Initialize default data for the new user

### Filter: `perfil-usuario/signup.redirect`

```php
$signup_redirect = 'dashboard/';

if (existe(gestor_sessao_variavel("redirecionar-local"))) {
    $signup_redirect = gestor_sessao_variavel("redirecionar-local");
    gestor_sessao_variavel_del("redirecionar-local");
}

$signup_redirect = hook_apply_filters('perfil-usuario', 'signup.redirect', $signup_redirect, $id_usuarios);

gestor_redirecionar($signup_redirect);
```

**Position in code:** Right before the final `gestor_redirecionar()` — the last operation before the HTTP redirect.

**Position in flow:**
```
Login → Token → Email
    ↓
    ★ hook_apply_filters('perfil-usuario', 'signup.redirect', $url, $id)
    ↓
gestor_redirecionar($url)
```

**Arguments:**

| Arg | Type | Description |
|-----|------|-------------|
| `$signup_redirect` | string | Current redirect URL |
| `$id_usuarios` | int | User ID |
| **Return** | string | URL (modified or not) |

**URL priority logic:**
1. If `redirecionar-local` session variable exists → use that URL (and clear it)
2. Otherwise → use `dashboard/`
3. The filter can override either of these

**Change from previous behavior:**

| Before (v0.2.0) | After (v0.3.0) |
|-----------------|-----------------|
| `if (existe(sessao)) { gestor_redirecionar(); } else { gestor_redirecionar('dashboard/'); }` | Resolve `$signup_redirect` → apply filter → `gestor_redirecionar($signup_redirect)` |

The `redirecionar-local` session variable is still respected — it's now resolved **before** the filter, which can optionally replace it.

## Design Decisions

### Why `gestor_incluir_biblioteca('hooks')` inside the function?

The `perfil-usuario` module may be loaded in contexts where the hooks library hasn't been included yet (e.g., public pages without admin modules). `gestor_incluir_biblioteca('hooks')` is **idempotent** — if already included, it has no effect.

### Why do plan data come from `$_REQUEST`, not the database?

At this point in the flow, the plan is merely an **intent** — the user selected a plan in the UI, but no plan record was created in the database (the plan/host blocks are disabled: `$desativado_planos = true`). The hook receiver (e.g., host-manager) is the one who validates and persists the information.

### Why `banco_escape_field` on the email?

The email already passed through `interface_validacao_campos_obrigatorios()` with the `email-obrigatorio` rule, but since we're passing data via array to third-party callbacks, the extra escaping ensures the data is sanitized against SQL injection in any downstream use.

---

## Compatibility

- **No registered hooks:** Behavior identical to before — `hook_do_action()` is a no-op, `hook_apply_filters()` returns the original value.
- **Missing hooks library:** If `gestor_incluir_biblioteca('hooks')` fails for any reason, the `hook_do_action()` and `hook_apply_filters()` functions won't exist and will cause a fatal error. Risk mitigated because the hooks library is part of the core and always available.
